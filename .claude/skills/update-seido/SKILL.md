# 制度データ更新スキル

1都3県212自治体の子育て支援制度データ（`src/prefs.json`）を、公式ソースのみで検証・更新する。

## モード

- **更新モード**（デフォルト）: 検証 → 差分レポート → ユーザー承認 → 反映 → デプロイ
- **チェックモード**（引数に `check`）: 反映せず変更検知のみ。4都県の公表ページを巡回し、`checked` 日以降の新公表・制度改定の兆候をレポートして終了。**ファイルは一切変更しない**

## スコープの決め方

引数から対象項目（待機児童・給食・医療費・経済的支援・全項目）と対象都県/自治体を読み取る。指定がなければ AskUserQuestion で確認する。全項目×全自治体の監査は大規模（数十万トークン）なので、明示的な依頼がない限り提案しない。

## 鉄則（データ品質）

1. **情報源は公式のみ**: 自治体公式サイト（`*.lg.jp` 等）と都県の公表資料（PDF/Excel）。ニュース記事・第三者まとめは不可
2. **推測で数値を作らない**: 確認できなければ `unverified` として報告。それらしい値の創作は厳禁
3. **mismatch判定は保守的に**: 明確に異なる公式値を確認できた場合のみ。年度違いの取り違えに注意（報告には必ず基準日を付ける）
4. **疑わしい値は独立再検証**: 人口規模との不整合（例: 島しょ部で待機児童2桁）や異例に早い公表年度は、別エージェントで一次資料を機械抽出（pdftotext / Excelパース）して確定させる。PDFの目視読みより機械可読データを優先
5. **国基準の値を採る**: 待機児童数は「隠れ待機児童」ではなく自治体が公表する国基準の値

## データ構造

`src/prefs.json`: `{"東京都": {"munis": [{...}], "prefItems": [...]}, ...}`（4都県・計212自治体）

自治体オブジェクトの主なフィールド:
- `hoiku`: 保育所待機児童数（人）/ `gakudo`: 放課後児童クラブ待機児童数
- `iryo`/`iryoChip`: 医療費助成の詳細/チップ表示 、`kyushoku`/`kyushokuChip`: 給食費
- `keizai`: 経済的支援、`birth`/`bday`/`tsuki`: 給付の金額データ（計算に使用）
- `refs`: `[{label, url}]` 出典リンク、`checked`: 確認日 `YYYY-MM-DD`

更新した自治体は必ず `checked` を当日に更新し、根拠URLを `refs` に追加する。

## 既知の公表パターン（待機児童数）

- **神奈川県**: 毎年6月下旬に県が独自公表（国の集計より早い）。別紙PDFに市町村別推移表
- **東京都**: 8月末「都内の保育サービスの状況」の表4（区市町村別）。**機械可読Excelがあるのでそれを優先**（metro.tokyo.lg.jp のdocuments配下）。世田谷・足立・練馬など一部区は個別に速報を出す
- **埼玉県**: 8月上旬に県PDF「市町村別保育所等待機児童数」（pref.saitama.lg.jp）
- **千葉県**: 8月頃に県PDF（pref.chiba.lg.jp、`r0X0401houdou.pdf` 形式）
- 公表時期のずれにより**基準年度が県ごとに混在**しうる。全県揃うのは9月頃。混在させた場合はUI注記（注5）に反映する
- 制度改定（医療費・給食・給付）は**4月の年度替わり**に集中する

詳細な経緯は memory の `taiki-jido-data-basis.md` を参照。

## 手順（更新モード）

1. スコープ確定後、対象自治体を scratchpad に 15〜20件/batch の JSON で書き出す（`{pref, name, 現在値, url}`）
2. batch ごとに general-purpose エージェントを**並列起動**。プロンプトには鉄則・出力先ファイル・出力形式（全件、`{name, current, verified, asof, source, status: match|mismatch|unverified, note}`）を明記する。エージェントには**孫エージェントを作らせない**（結果ファイルを書かずに停止しがち。起きたら親をSendMessageで再開させるか、子の報告から自分で結果を再構成する）
3. 集約して mismatch / null補完 を抽出。疑わしい値（鉄則4）は再検証エージェントで確定
4. **差分レポートをユーザーに提示**（自治体名、旧→新、基準日、出典URL）し、承認を待つ
5. 承認後に反映:
   - 事前に PREFS の JSON round-trip 一致（`json.dumps(json.loads(raw))==raw` 相当）を確認してから、Python で一括更新
   - 反映後、変化したフィールドが意図したもの（対象フィールド＋`checked`＋`refs`）だけかを git diff ベースで検証
6. `npm test` → プレビューで表示確認（データはAPI `/api/prefs` 経由。`public/index.html` はビルド生成物なので直接編集しない、ソースは `app/index.html`）
7. 注記の基準日表記（`app/index.html` 注5）と memory の年度情報を必要に応じて更新
8. ユーザーに確認のうえ コミット → `npm run deploy` → 本番で `curl` により反映確認（ブラウザ確認はキャッシュに注意。オリジン確認はcurlが確実）

## 手順（チェックモード）

1. `src/prefs.json` の `checked` 分布と現在日から、確認が古い領域を把握
2. 4都県の公表ページ（上記パターンのURL群）を WebFetch で確認し、新しい公表・改定告知がないか調べる
3. 数自治体をサンプル抽出して公式ページの更新を確認（4月・9月は多め）
4. 「変更が出ている可能性のある項目と自治体」をレポート。更新の実行はユーザーが `/update-seido <スコープ>` で判断する
